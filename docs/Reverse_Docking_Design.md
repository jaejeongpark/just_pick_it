# Reverse Docking 정밀 후진 도킹 설계

> ArUco 마커 + 노란 주차라인을 결합한 차동구동 AMR의 정밀 후진 도킹 시스템 설계 기록.
> **설계 고찰 → 방식과 선택 → 검증**의 흐름으로, 의사결정의 근거를 중심으로 정리한다.

---

## 1. 문제 정의와 요구사항

자율주행 AMR(PICKY)이 충전 도크에 **후진으로** 진입한다. 도크가 **매우 좁아**(로봇 폭에 여유가 거의 없음) 최종 정렬 정밀도가 시스템 성공/실패를 가른다.

| 구분 | 내용 |
|---|---|
| **목표** | 좁은 도크에 충돌 없이 정밀 후진 삽입 + 도킹 후 위치추정(AMCL) 재보정 |
| **센서** | 전방 단일 카메라(캘리브레이션 완료), ArUco 마커 1개/도크, 바닥 노란 주차라인 |
| **로봇** | 차동구동(비홀로노믹: 횡방향 직접 이동 불가), 헤드(+x_body)가 +y_world를 보며 -y로 후진 |
| **좌표계** | `map` 프레임 = zone 절대 프레임(좌하단 원점, `origin=[0,0,0]`). 마커·도크 월드좌표 기지 |

**정밀도 정량 목표 (기하로 역산)**

후진 거리 ≈ 0.33 m (standby `y≈0.40` → dock `y≈0.07`). 최종 횡오차를 **1 cm 이내**로 두려면

```
lateral_error ≈ reverse_distance · sin(yaw_error)
0.01 m ≥ 0.33 m · sin(yaw_error)   ⇒   yaw_error ≲ 1.7°
```

즉 **yaw 정밀도 ~1°**가 핵심 제약이다. 이 한 줄이 이후 모든 설계 선택을 지배한다.

## 2. 설계 고찰 — 정밀도의 병목은 어디인가

후보 센싱을 정밀도 관점에서 분해했다.

### 2.1 단일 평면 마커 solvePnP의 한계

| 측정 축 | 정밀도 | 근거 |
|---|---|---|
| **Lateral (횡)** | **서브밀리** | 마커가 정면 → 픽셀 오프셋이 곧 횡오차. `fx≈985`, 0.5 m에서 **1 px ≈ 0.5 mm** |
| Depth (깊이) | 양호 | 마커 크기/거리 관계로 안정적 |
| **Yaw** | **취약 (1~3°)** | 평면 마커 법선 추정 노이즈 + **pose-flip ambiguity** |

→ **횡은 쉽고, yaw가 어렵다.** 그런데 좁은 도크의 정밀도 제약은 바로 그 yaw(≲1.7°)다. **단일 마커 solvePnP yaw만으로는 마진이 부족**하다.

### 2.2 발상 전환 — yaw는 다른 센서로

바닥의 **두 노란 주차라인**은 도킹 채널을 직접 정의한다. 두 라인의 **대칭선(채널 중심)**을 이미지 중심·수직(법선)에 맞추는 lane-keeping은:

- 픽셀 기반이라 캘리브레이션 오차에 둔감
- 평면 마커 yaw ambiguity를 **원천 회피**
- 후진할수록 라인이 더 많이 잡혀 정밀도가 **오히려 향상**

즉 **정밀도가 필요한 마지막 구간일수록 라인 신호가 좋아진다** — 문제와 해법의 방향이 일치한다.

## 3. 방식과 선택 — 대안 비교와 결정

### 3.1 제어 패러다임 비교

| 방식 | 설명 | 평가 |
|---|---|---|
| **상대 시각서보 (구버전)** | 마커 tvec/rvec에 PID, "마커에 가까워질 때까지 후진" | ❌ 마커가 도크 뒤에 있다고 **가정**. 새 배치(마커는 벽 위, 도크는 아래)에선 후진 시 마커에서 **멀어져** 전제 붕괴 |
| **PBVS 월드프레임** | 마커→로봇 월드 pose 계산 후 도크 pose로 주행 | △ 일반성·정확성 좋으나 **yaw 정밀도가 solvePnP에 종속** (= 2.1의 병목) |
| **IBVS** | 이미지 특징 직접 서보 | △ 캘리브 오차엔 강하나, 마커-도크 **x오프셋(4 cm)**을 픽셀 타깃으로 환산해야 해 복잡. 기지 월드좌표의 이점도 미활용 |
| **✅ 하이브리드 (채택)** | **마커=깊이·coarse·보정, 라인=정밀 lateral+yaw** | 각 센서를 **강점 축에만** 사용. yaw를 라인으로 빼 병목 제거 |

### 3.2 채택 설계 — 역할 분담

> **핵심 원칙: 각 센서를 가장 정밀한 축에만 쓴다.**

| 축 | 담당 | 방식 |
|---|---|---|
| **lateral + yaw** (정밀 핵심) | **노란 주차라인** | 두 라인 → 대칭선 → (수평오프셋 + 기울기) → PID |
| **depth / 정지 시점** | **ArUco 마커** | 마커 거리로 로봇 world y 추정 → `dock_y` 도달 시 정지 |
| **시작 coarse 정렬 + 위치 보정** | **ArUco 마커** | 라인 부족 구간 coarse 정렬, 도킹 후 `/initialpose` |

### 3.3 가시성 그라데이션과 신뢰도 블렌딩

라인은 **처음엔 일부만, 후진할수록 더 많이** 잡힌다. 하드 페이즈 전환 대신 **라인 검출 신뢰도(conf)로 연속 블렌딩**:

```
ω = conf · (라인 PID: lateral+yaw 합산)  +  (1 − conf) · (마커 coarse 정렬)

  시작 (conf→0)        삽입 (conf→1)
  ┌──────────────┐     ┌──────────────┐
  │ 마커 coarse   │ ──▶ │ 라인 정밀     │   매끄러운 전환, ω 단일값(부드러운 움직임)
  └──────────────┘     └──────────────┘
```

`conf`는 양쪽 라인 픽셀 수로 0~1. 시작 구간은 마커가 채널 진입을 책임지고(Nav2로 이미 대략 정렬되어 도착), 정밀이 필요한 삽입 구간은 라인이 전담한다.

**파생 결정**
- **벽 라인 불필요**: 정밀도가 필요한 구간엔 바닥 라인이 충분히 보이고, 시작은 마커 coarse로 족하다. (테스트에서 시작 드리프트가 보이면 마커 양옆 벽 라인으로 보강 — 쉬운 업그레이드라 선제 적용 안 함)
- **단일 마커로 충분**: yaw를 라인이 담당하므로 마커 보드(다중 태그)까지 갈 필요 없음
- **오도메트리 융합 불필요**: 라인이 절대 채널 기준을 매 프레임 제공

### 3.4 기하 구성

```
   y=0.655  ┌───────[ ArUco ]───────┐  ← 가로벽, 마커 법선 −y(도크 쪽)
            │         │  │          │
            │     노란 │  │ 주차라인  │  ← 후진할수록 더 많이 보임
            │         │  │          │
            │       [로봇] ↑ 카메라(+y)│
            │         ▲ 후진(−y)      │
   y=0.07   └────────[도크]──────────┘
            x: dock1 marker 0.07 / dock 0.11  (4cm 오프셋 → coarse 횡 목표에 반영)
                dock2 marker 0.28 / dock 0.28  (정렬)
```

### 3.5 제어 법칙

```
ω (각속도) = 블렌딩된 정렬 오차  →  lateral·yaw를 한 값으로 합쳐 부드럽게
v (병진)   = −reverse_speed (느린 등속 후진)
정지       = robot_y(= marker_world_y − 마커거리 − 카메라전방오프셋) ≤ dock_y
보정       = 도크 절대좌표를 /initialpose 로 발행 (AMCL 재초기화)
```

차동구동이라 lateral·yaw 둘 다 ω로 표현되며, 합산이 분리 제어보다 **움직임이 매끄럽다**.

### 3.6 모듈 아키텍처

```
ReverseDocking (node)
├─ MarkerLocalizer : ArUco 검출 + solvePnP(IPPE_SQUARE) → 깊이 / coarse 정렬 / 보정
├─ LaneDetector    : 노란라인 2개 직선피팅 → 대칭선 (수평오프셋, 기울기, 신뢰도)
├─ DockController  : ω = blend(라인PID, 마커coarse),  v = 후진,  깊이로 정지
└─ reverse_dock()  : A 마커획득 → C 후진삽입(블렌딩) → D 정지+보정
```
- 인터페이스 `reverse_dock(marker_id, dock_x, dock_y, dock_yaw)`는 **불변** → 상위 state_manager 무수정
- 마커 월드좌표는 reverse_docking 설정에 per-`marker_id`로 보관(전역 zone DB 미오염)
- 카메라 extrinsic은 URDF/TF에서 조회(하드코딩 배제)

## 4. 구현 세부 — 정밀도를 위한 선택들

| 항목 | 선택 | 이유 |
|---|---|---|
| solvePnP solver | `SOLVEPNP_IPPE_SQUARE` | 평면 정사각 마커 전용, **ambiguity 처리** 우수 |
| 카메라 intrinsic | 비전팀 캘리브레이션 실측값 (per-robot) | placeholder 제거. ROS `camera_info` → 평탄 9+5 포맷으로 변환 적용 |
| 마커-도크 x오프셋 | coarse 횡 목표 = `marker_x − dock_x` | 마커가 도크 바로 위가 아니어도(예: 4 cm) 정확히 정렬 |
| 라인 yaw | 대칭선 기울기(`atan2(vx, |vy|)`) | 평면 마커 yaw 대신 라인 기울기로 — 노이즈·ambiguity 회피 |
| 블렌딩 신뢰도 | 양쪽 라인 최소 픽셀 수 기반 0~1 | 가시성 그라데이션을 연속 가중 |
| 비상 정지 | 공유 EmergencyLatch, pause-continue | 도킹 중 비상 정지 시 제자리 정지 후 동일 동작 재개 |

> 캘리브레이션 해상도(1280×720)와 `camera/image_raw` 발행 해상도가 일치해야 intrinsic이 유효 — 설정에 명시.

## 5. 검증

### 5.1 정적 검증 (완료)

| 검증 | 방법 | 결과 |
|---|---|---|
| 캘리브레이션 정합 | 적용값 ↔ 캘리브 원본 비교 | ✅ camera_matrix(9)·dist(5) 정확히 일치, DOUBLE_ARRAY 파싱 |
| 코드 문법 | `py_compile` | ✅ 통과 |
| 설정 파싱 | YAML 로드 + 파라미터 키 검증 | ✅ 24개 파라미터, 마커 월드좌표 정상 |
| API 가용성 | `SOLVEPNP_IPPE_SQUARE`, `ArucoDetector` | ✅ (ArucoDetector는 보드 cv≥4.7 대상) |
| 인터페이스 호환 | `reverse_dock()` 시그니처 불변 | ✅ state_manager 무수정 |

### 5.2 하드웨어 브링업 검증 (계획)

설계상 **부호·게인은 실차 튜닝 대상**(원 구현도 placeholder였음). 단계별 검증 계획:

1. **마커 검출/거리** — 정지 상태에서 marker 거리 vs 실측 거리 일치 확인, IPPE yaw 안정성 관찰
2. **라인 검출** — 노란 HSV 조명 튜닝, 대칭선 오프셋/기울기가 실제 자세와 부호 일치 확인
3. **부호·게인** — coarse/라인 ω 부호 확정, `lane_lat_kp`/`lane_yaw_kp` 점진 튜닝(진동 없는 임계까지)
4. **End-to-End 정밀도** — 반복 도킹 후 **최종 lateral·yaw 오차 측정**:
   - 목표: lateral ≤ 1 cm, yaw ≤ 1.5° (§1 역산 기준)
   - 지표: 평균/표준편차/최악값, 반복성(repeatability)
5. **위치 보정 정확도** — 도킹 후 `/initialpose` 발행 → AMCL 수렴 위치 vs 실제 도크 오차

### 5.3 정밀도 검증 지표(설계)

```
성공 기준
  lateral_error  ≤ 10 mm   (좁은 도크 마진)
  yaw_error      ≤ 1.5°    (0.33 m 후진 시 횡 1cm 환산)
  repeatability  σ_lateral ≤ 5 mm  (N회 반복)
  pose_correction_error ≤ 3 cm (AMCL 재초기화 정합)
```

## 6. 설계 요약

| 결정 | 근거 |
|---|---|
| 마커=깊이·보정, 라인=정밀 lateral+yaw | 정밀도 병목(yaw)을 ambiguity 없는 라인으로 분리 |
| 신뢰도 기반 마커→라인 블렌딩 | 가시성 그라데이션을 끊김 없이 흡수 |
| 단일 마커 + 벽 라인 미사용 | yaw를 라인이 담당 → 보드/벽 라인 없이도 충분 |
| 월드프레임 깊이 정지 | 정렬은 라인, 정지는 마커 월드 기준 → 각 축 최적 |
| state_manager 무수정 | 도킹 내부 재설계를 상위에 격리 |

**핵심 한 줄**: *정밀도 제약(yaw ≲1.7°)을 기하로 먼저 역산하고, 각 센서를 가장 정밀한 축에만 배치해 병목을 구조적으로 제거했다.*

## 7. 실차 브링업 — 설계 개정 기록 (2026-06-11)

> §1~6은 책상 위 설계다. 실차에 올리자 **두 전제가 무너졌고**, 그 결과 도킹 제어를
> 거의 새로 짰다. 이 절은 "무엇이 깨졌고, 왜 이렇게 바꿨고, 수치가 어떻게 변했나"의 기록이다.

### 7.0 무너진 두 전제

| 원설계 전제 | 실차에서 깨진 이유 |
|---|---|
| **라인이 lateral+yaw 정밀을 담당** | standby 거리에선 **주차라인이 카메라에 안 잡힌다**(가까워져야 보임). 잡혀도 한 줄만/conf 낮음 → 대칭선 신뢰 불가 |
| **마커 solvePnP로 실시간 서보** | 마커 정면(rvec_y≈±π)에서 **rvec 부호가 프레임마다 ±로 뒤집히는 ambiguity** + psi(yaw)가 정지 상태에서도 ±5~8° 출렁 → 실시간 robot_x/yaw 제어가 발산 |

→ **결론**: 라인은 마지막 구간 보조로 강등(기본 off), **마커는 "실시간 서보"가 아니라 "1회 측정 기준점"으로만** 쓰고, 실제 이동·헤딩 유지는 **노이즈 없는 odom**에 맡긴다.

### 7.1 근본원인 추적 — 증상 → 원인 → 해결

검출/정렬 파이프라인을 한 겹씩 벗겨 같은 증상의 서로 다른 원인을 순차 격리했다.

| # | 증상 | 근본 원인 | 해결 |
|---|---|---|---|
| 1 | 30s acquire timeout(마커 0회) | 보드에 `camera/image_raw` 발행자 없음 | reverse_docking 이 **Picamera2 직접 캡처**(도킹 중에만) |
| 2 | 카메라 켜도 미검출 | 도크 마커는 **AprilTag 36h11** 인데 코드는 ArUco 4x4_50 | dict 정정 + 캘리브 yaml 직접 로드 |
| 3 | 깊이 정지시점 어긋남 | marker_size 0.10 인데 실측 **0.05** | size 0.05 |
| 4 | 정렬이 반대로 → 벽 충돌 | **카메라 거꾸로 장착 + flip 캘리브**인데 flip 미적용 | 검출 전 `cv2.flip(-1)` |
| 5 | 정렬 위치 x 어긋남 | dock1 marker_world_x 0.07 가정(실제 dock1/standby1/마커 **일직선 x=0.11**) | marker_world_x 0.11, marker_world_y 0.655→0.635 |
| 6 | 깊이 1.48배 짧게 | solvePnP tvec[z] 가 실측의 1/1.48 | `depth_scale` 도입(자=0.175 vs tvec=0.118) |
| 7 | **횡 제어가 발산**(법선 지나쳐 아크로 멀어짐) | 실시간 `robot_x` 가 **노이즈 psi**에 종속 — 회전만 해도 robot_x 가 수 cm 헛돔(부호로 해결 불가) | 실시간 추적 폐기 → **1회 측정 + odom arc** |
| 8 | 측정한 robot_x 가 부정확(조준≠법선) | 마커 "조준"(tvec_x→0)은 헤딩이 틀어진 채로도 성립 → robot_x 가 헤딩과 결합 | psi 디커플링 후 **측정 전 똑바로 세움**(psi≈0 → Δx 깨끗) |
| 9 | 횡오차 1.48배 과소 | Δx 계산에 depth_scale 누락(깊이엔 적용했으나 횡엔 raw) | Δx 에도 스케일 적용(→ 횡 전용 `lateral_scale`) |
| 10 | 도크 4cm 앞 조기 정지 | depth_scale 가 **가까운 거리(standby)에서 튜닝** → 원거리(dock)에서 과대 | 깊이/횡 스케일 분리: depth 1.36(원거리) / lateral 1.48(가까움) |
| 11 | 후진 중 yaw 누적 드리프트 | 작은 yaw 오차 × 후진거리 = x 드리프트. psi 는 노이즈라 후진 중 못 씀 | **odom yaw 앵커(θ_ref) + 후진 내내 유지** |
| 12 | recenter 회전이 반대 | centering 부호 `+`(마커 왼쪽인데 오른쪽으로 회전) | `omega=-recenter_kp·bearing` (정적검증 기준) |

> **keystone 두 개**: (4) 물리적으로 거꾸로 달린 카메라, 그리고 (7) **rvec yaw 의 구조적 노이즈**. 후자가 "라인+실시간 마커" 설계 전체를 "마커 1회 + odom" 으로 뒤집었다.

### 7.2 개정 설계 — 마커 1회 측정 + odom 실행

```
acquire(마커 검출)
 → 초기 recenter(마커 중앙)
 → [ 똑바로 세움(psi→0) → Δx 측정(깨끗) → 후진 arc(odom 으로 |Δx| 이동) → recenter ] × 2
 → 똑바로 세움 → odom yaw θ_ref 앵커
 → 직진 후진(odom yaw=θ_ref 유지) → 마커 깊이로 정지 → /initialpose 보정
```

| 역할 | 담당 (개정) | 왜 |
|---|---|---|
| 횡 위치(법선 진입) | **마커 1회 측정 + odom arc** | psi 노이즈를 측정 1회로 격리, 이동량은 odom(정밀·평활)으로 |
| 헤딩 유지(후진) | **odom yaw 앵커 θ_ref** | psi 는 후진 중 못 씀 → 마커로 법선 1회 잡고 odom 으로 정밀 유지 |
| 깊이/정지 | ArUco 마커 (원거리 scale) | 월드 깊이 `robot_y ≤ dock_y` |
| 라인 정밀 | 기본 off (`use_lane_steering`) | 원거리 불검출 → 토글만 유지(품질 개선 시 on) |

**핵심 통찰 3가지**
1. **노이즈 큰 센서(rvec/psi)는 "1회·정지·평균"으로만 쓴다.** 매 프레임 추적하면 노이즈가 제어로 들어와 발산.
2. **측정 전 똑바로 세우면 측정이 깨끗해진다.** psi≈0 이면 `Δx ≈ −tx·scale`(tx 는 저노이즈) → phantom 제거.
3. **odom 은 전역 누적오차가 문제지, 짧은 상대이동/헤딩 유지엔 가장 정밀하다.** "cmd_vel 을 믿는다"의 가장 정확한 구현이 odom.

### 7.6 남은 과제

- **정렬 정밀도 floor ~1cm / yaw ~2°**: 단일 마커 rvec 의 노이즈 한계. 더 줄이려면 측정 평균 프레임↑ 또는 마커 보드(다중 태그).
- **보드 CPU 과부하**: 도킹 비전 루프가 Pi 한 코어 포화 → 주기↓/경량화 필요.
- **N회 반복 정량화**: final x · 남은 깊이 · yaw 를 5회+ 측정해 §1 목표(lateral≤1cm, yaw≤1.5°) 대비 채우기.
