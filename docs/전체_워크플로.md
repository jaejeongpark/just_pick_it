# Just Pick It 전체 워크플로

이 문서는 현재 구현 기준으로 Customer/Admin Web UI에서 요청이 들어온 뒤
Fleet Manager가 어떤 순서로 DB를 바꾸고 task를 자동 생성/실행하는지 정리한다.

## 현재 실행 구조

```text
Customer/Admin Browser
  -> Web Gateway (:8000)
  -> Fleet API (:8100)
  -> FleetRepository
  -> just_pick_it_db
  -> PostgreSQL

FleetManagerNode
  -> TaskManager
  -> TrafficManager
  -> RobotCommandGateway
  -> PICKY/COBOT State Manager
```

핵심 원칙:

```text
- Web Gateway는 화면과 proxy만 담당한다.
- Fleet API는 Fleet Manager 프로세스 안에서 뜬다.
- DB 접근은 FleetRepository가 담당한다.
- TaskManager가 주문/입고를 polling해서 task를 자동 생성한다.
- TrafficManager는 PICKY 경로 예약/해제를 담당한다.
- RobotCommandGateway는 ROS2 Action/Service 명령 송신을 담당한다.
```

## 실행 시작 흐름

```text
./run_all.sh
  -> PostgreSQL 준비 확인
  -> ros2 launch fleet_manager fleet_manager.launch.xml
  -> FleetManagerNode 시작
  -> Fleet API :8100 시작
  -> Web Gateway :8000 시작
```

Fleet Manager 내부 생성 순서:

```text
FleetManagerNode
  -> FleetRepository 생성
  -> RobotCommandGateway 생성
  -> FleetRepository.fetch_zone_coords()
  -> TrafficManager 생성
  -> RobotStateMonitor 생성
  -> TaskManager 생성
  -> 5초 polling timer 생성
  -> FleetApiServer 시작
```

설정 파일:

```text
src/just_pick_it/fleet_manager/config/fleet_manager.yaml
```

현재 기본값:

```text
api_port: 8100
api_push_interval_sec: 1.0
waiting_work_poll_period_sec: 5.0
robots: PICKY1, PICKY2, COBOT1, COBOT2
```

## 5초 polling의 정확한 의미

현재 구현은 "무조건 5초마다 주문/입고를 가져온다"가 아니라,
"5초마다 확인하되, 새 작업을 받을 수 있는 PICKY가 있을 때만 주문/입고 polling을 연다"가 정확하다.

```text
FleetManagerNode.create_timer(5.0, _poll_waiting_work_if_picky_idle)
  -> TaskManager.has_idle_picky_for_waiting_work()
  -> true면 TaskManager.check_waiting_work()
```

`has_idle_picky_for_waiting_work()` 통과 조건:

```text
- fleet emergency pause 상태가 아님
- PICKY robot_status == IDLE
- PICKY picky_state == STANDBY 또는 None
- PICKY current_task_id 없음
- 해당 PICKY에 열린 task 없음
- LOW_BATTERY housekeeping 진행 중 아님
- battery_level이 있으면 40보다 큼
```

`check_waiting_work()`가 하는 일:

```text
1. 완료 가능한 CHARGE task 정리
2. 신규 ORDER_WAIT 주문 / REQUESTED 입고 polling
3. 기존 주문 flow에서 다음 task 생성 가능한지 확인
4. 기존 입고 flow에서 다음 task 생성 가능한지 확인
5. ASSIGNED task 중 실행 가능한 task dispatch
```

정상 task 진행은 다음 polling까지 기다리는 구조가 아니다.
Action result가 오면 `handle_task_result()`에서 즉시 다음 단계로 넘어간다.
5초 polling은 새 작업과 막혀 있던 흐름을 다시 확인하는 보정/진입점에 가깝다.

## Customer 주문 생성 흐름

```text
Customer Browser
  -> POST /api/orders
  -> Web Gateway proxy
  -> Fleet API POST /api/orders
  -> FleetRepository.create_order()
```

DB 변경:

```text
product.stock_qty 차감
orders row 생성
order_item row 생성
order.status = ORDER_WAIT
```

이 시점에는 task가 아직 없다.
주문은 DB에 `ORDER_WAIT` 상태로 들어가고, Fleet Manager polling이 가져간다.

## 주문 자동 task 생성 흐름

```text
5초 timer
  -> PICKY IDLE/STANDBY 확인
  -> TaskManager.check_waiting_work()
  -> FleetRepository.list_waiting_orders()
  -> TaskManager._process_new_order(order)
```

`_process_new_order()` 흐름:

```text
1. _select_available_unit()
   - 같은 unit_id의 PICKY/COBOT pair 선택
   - 둘 다 작업 가능해야 함
   - 후보가 여러 개면 PICKY battery_level 높은 unit 우선

2. FleetRepository.update_order_status()
   - assigned_unit_id 기록

3. FleetRepository.get_order_work()
   - 주문 item, 상품 zone/slot, robot 이름을 task 생성용 dict로 정규화

4. _create_next_product_tasks()
   - 남은 상품 zone 후보를 만든다
   - TrafficManager.reserve_nearest_from()으로 가장 가까운 PRODUCT_ZONE 예약
   - MOVE_TO_PRODUCT / SORTING_AND_LOAD task 2개 생성
   - TrafficManager.attach_task_id()로 임시 예약을 실제 move task_id에 연결

5. _dispatch_ready_tasks()
   - 실행 가능한 ASSIGNED task를 로봇으로 보낸다
```

첫 상품 task:

```text
sequence 1: PICKY  MOVE_TO_PRODUCT
sequence 2: COBOT  SORTING_AND_LOAD
```

## 주문 task 실행 흐름

### PICKY 이동 task

```text
TaskManager._dispatch_move_task()
  -> TrafficManager 경로 예약 확인
  -> FleetRepository.update_task_status(RUNNING)
  -> RobotCommandGateway.send_move_task()
  -> /pickyX/move_command Action goal
```

Action feedback:

```text
PICKY MoveCommand feedback
  -> RobotCommandGateway
  -> TaskManager.handle_move_feedback()
  -> TrafficManager.update_path_progress()
```

Action result:

```text
PICKY MoveCommand result
  -> RobotCommandGateway
  -> TaskManager.handle_task_result()
  -> FleetRepository.update_task_status(SUCCESS 또는 FAILED)
  -> TrafficManager.release_path()
  -> 성공이면 다음 task dispatch
```

PICKY action server가 없거나 waypoint pose 변환에 실패하면:

```text
task.status = FAILED
exception_log 생성
TrafficManager path release
```

### COBOT 작업 task

주문에서 COBOT task는 현재 다음 종류다.

```text
SORTING_AND_LOAD
INSPECTION
UNLOAD
```

현재 저장소에는 COBOT `ExecuteTask.action`이 아직 정의되어 있지 않다.
그래서 `RobotCommandGateway.send_cobot_task()`는 실제 action goal을 보내지 않고 false를 반환한다.

현재 동작:

```text
TaskManager._dispatch_cobot_task()
  -> RobotCommandGateway.send_cobot_task()
  -> ExecuteTask.action 정의 대기
  -> task는 ASSIGNED 상태 유지
```

즉, 현재 로컬에서 주문 생성과 PICKY MOVE dispatch까지는 확인 가능하지만,
COBOT task 성공까지 자동 진행하려면 COBOT action 정의/서버가 필요하다.

## 주문 다음 상품 / pickup task 생성

task result가 SUCCESS로 들어오면 polling을 기다리지 않고 즉시 흐름을 이어간다.

```text
TaskManager.handle_task_result()
  -> _handle_task_result_locked()
  -> _advance_flow_after_task_success()
  -> _advance_order_by_id_if_ready()
```

주문 flow 규칙:

```text
현재 생성된 주문 task가 모두 SUCCESS
  -> WAITING order_item이 남아 있으면 다음 상품 task 생성
  -> WAITING order_item이 없으면 pickup task 생성
  -> pickup task까지 모두 끝나면 RETURN_HOME/DOCK_IN/CHARGE housekeeping 생성
```

다음 상품 task:

```text
MOVE_TO_PRODUCT
SORTING_AND_LOAD
```

pickup task:

```text
MOVE_TO_PICKUP
INSPECTION
UNLOAD
```

pickup task 생성 시:

```text
1. FleetRepository.list_pickup_slots(status="EMPTY")
2. PICKUP_SLOT_N -> PICKUP_ZONE_N 변환
3. TrafficManager.reserve_nearest_from()으로 가까운 pickup zone 선택
4. FleetRepository.update_order_status(pickup_slot_id=...)
5. MOVE_TO_PICKUP / INSPECTION / UNLOAD task 생성
6. MOVE_TO_PICKUP 예약을 task_id에 attach
```

`UNLOAD`가 SUCCESS가 되면:

```text
order.status = PICKUP_READY
pickup_slot.status = OCCUPIED
```

Customer가 수령 완료 버튼을 누르면:

```text
POST /api/orders/{order_id}/complete
  -> order.status = COMPLETED
  -> pickup_slot.status = EMPTY
```

## Admin / LLM 입고 요청 흐름

현재 입고 요청은 `stocking_item`을 만드는 방식이다.

Admin AI 버튼 흐름:

```text
Admin Browser
  -> POST /api/admin/llm/messages
  -> Web Gateway llm_router
  -> web/app/services/llm_client.py
```

현재 `llm_client.py`는 stub이다.
담당자가 실제 parser를 구현해서 아래 형태를 반환해야 한다.

```text
action = STOCKING
product_id
requested_quantity
stocking_policy
```

`action=STOCKING`으로 파싱되면:

```text
llm_router
  -> Fleet API POST /api/admin/stocking-items
  -> FleetRepository.create_stocking_item()
```

DB 변경:

```text
stocking_item row 생성
stocking_item.status = REQUESTED
```

이 시점에도 task는 아직 없다.
입고 요청은 `REQUESTED` 상태로 들어가고, Fleet Manager polling이 가져간다.

## 입고 자동 task 생성 흐름

```text
5초 timer
  -> PICKY IDLE/STANDBY 확인
  -> TaskManager.check_waiting_work()
  -> FleetRepository.list_requested_stocking_items()
  -> TaskManager._process_new_stocking_item(stocking_item)
```

`_process_new_stocking_item()` 흐름:

```text
1. _select_available_unit()
   - PICKY/COBOT pair 선택

2. FleetRepository.get_stocking_work()
   - STOCK_ZONE/STOCK_SLOT
   - 상품 보관 PRODUCT_ZONE/PRODUCT_SLOT
   - robot 이름을 task 생성용 dict로 정규화

3. create_stocking_tasks_for_item()
   - 입고 task 4개 생성

4. FleetRepository.update_stocking_item()
   - status = ASSIGNED
   - assigned_unit_id 기록

5. _dispatch_ready_tasks()
   - sequence 1부터 실행
```

입고 task:

```text
sequence 1: PICKY  MOVE_TO_STOCK
sequence 2: COBOT  STOCKING_PICK
sequence 3: PICKY  MOVE_TO_STORAGE
sequence 4: COBOT  STOCKING_PLACE
```

주문 첫 상품과 다른 점:

```text
- 주문 MOVE_TO_PRODUCT는 task 생성 전에 TrafficManager로 먼저 예약한다.
- 입고 MOVE_TO_STOCK/MOVE_TO_STORAGE는 task 생성 시점이 아니라 dispatch 시점에 경로를 예약한다.
```

`STOCKING_PLACE`가 SUCCESS가 되면:

```text
stocking_item 완료 처리
product.stock_qty 반영
```

현재 실제 완료에는 COBOT action/result 연동이 필요하다.

## 주문과 입고가 같이 대기 중일 때

TaskManager는 주문과 입고를 하나의 대기열로 모은다.

```text
FleetRepository.list_waiting_orders()
FleetRepository.list_requested_stocking_items()
  -> WorkRequest 리스트 생성
  -> priority, work_id, kind 순으로 정렬
```

정렬 기준:

```text
1. priority 낮은 값 우선
2. work_id 낮은 값 우선
3. kind 문자열 순서
```

실제 배정 직전에도 `_select_available_unit()`이 다시 robot unit 사용 가능 여부를 확인한다.
따라서 같은 polling cycle에서 같은 unit에 주문과 입고가 중복 배정되지 않도록 open task를 다시 검사한다.

## 상태가 Web UI에 반영되는 흐름

읽기 API:

```text
Admin UI
  -> GET /api/admin/status
  -> FleetRepository.get_snapshot()

Customer UI
  -> GET /api/customer/status
  -> FleetRepository.get_customer_snapshot()
```

WebSocket:

```text
Admin UI
  -> WS /api/admin/ws/status
  -> Web Gateway proxy
  -> Fleet API /api/admin/ws/status
  -> FleetRepository.get_snapshot()
  -> 1초마다 push

Customer UI
  -> WS /api/customer/ws/status
  -> Web Gateway proxy
  -> Fleet API /api/customer/ws/status
  -> FleetRepository.get_customer_snapshot()
  -> 1초마다 push
```

즉, task 상태가 바뀌면 DB가 바뀌고, Fleet API WebSocket snapshot이 UI에 반영한다.

## Robot 상태 반영

현재 구현:

```text
PICKY State Manager
  -> /pickyX/picky_state topic
  -> RobotStateMonitor
  -> TrafficManager.notify_state()
```

Task RUNNING/SUCCESS/FAILED에 따른 DB robot 상태 변경은
`FleetRepository.update_task_status()` 내부에서 `apply_task_runtime_state()`를 통해 처리된다.

아직 확장 대기:

```text
- battery topic -> TaskManager.handle_battery_update()
- pose topic -> FleetRepository.update_robot_state()
- COBOT state/STOWING_ARM feedback -> TaskManager.preplan_after_cobot_stowing()
```

## Emergency / Resume 흐름

```text
Admin Browser
  -> POST /api/admin/emergency-stop 또는 /api/admin/resume
  -> Web Gateway proxy
  -> Fleet API
  -> FleetManagerNode.trigger_emergency_stop()
```

Emergency stop:

```text
FleetRepository.apply_emergency_stop()
  -> 모든 robot.robot_status = EMERGENCY_STOP
  -> RUNNING task = PAUSED
  -> TaskEvent 기록

RobotCommandGateway.set_emergency_stop(true)
  -> /robot_ns/emergency_control service 호출

TaskManager.handle_emergency_stop()
  -> 신규 polling/dispatch gate 닫음
```

Resume:

```text
FleetRepository.apply_resume()
  -> EMERGENCY_STOP robot 정리
  -> PAUSED task를 RUNNING으로 복구할 수 있으면 복구

RobotCommandGateway.set_emergency_stop(false)
  -> /robot_ns/emergency_control service 호출

TaskManager.handle_resume()
  -> gate 열고 가능한 flow/dispatch 재시도
```

## 현재 구현 한계

현재 구현상 전체 task 성공까지 보려면 아래 서버가 필요하다.

```text
- PICKY MoveCommand action server
- PICKY DockCommand action server
- COBOT ExecuteTask.action 정의 및 action server
- 각 robot EmergencyControl service server
```

없는 경우:

```text
- 주문/입고 요청 생성 가능
- 자동 task 생성 가능
- PICKY action server가 없으면 MOVE task는 dispatch 시 FAILED 처리
- COBOT action이 없으면 COBOT task는 ASSIGNED 상태로 유지
```

## 한 줄 요약

```text
Web은 요청/표시만 한다.
Fleet API는 Fleet Manager 안에서 DB 접근을 열어준다.
FleetRepository가 DB를 바꾼다.
TaskManager가 5초 polling과 action result를 기준으로 task를 자동 생성/진행한다.
TrafficManager가 PICKY 경로를 예약한다.
RobotCommandGateway가 실제 로봇 Action/Service를 호출한다.
```
