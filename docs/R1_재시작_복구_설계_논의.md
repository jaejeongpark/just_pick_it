# R1 — Fleet Manager 재시작 시 RUNNING task 복구 설계 (논의용)

작성일: 2026-05-28
상태: **논의 중** (담당: 박서우 = Repository/Traffic/State Manager, 이명제 = Task Manager)
관련: `docs/Fleet_Manager_분담_및_TODO.md`의 [R1]

이 문서는 "Fleet Manager가 RUNNING task 도중 재시작되면 무엇이 깨지고, 어떻게 복구할지"에 대한
검토 내용을 팀 상의용으로 정리한 것이다. 마지막 §8의 "함께 정할 것"을 보고 결정하면 된다.

---

## 1. 배경 / 문제

`MultiThreadedExecutor`로 노드가 재기동되면 **in-memory 상태가 전부 초기화**된다.

- TrafficManager: `_robot_paths` / `_robot_reservations` / `_robot_dock` (예약·점유) 가 빈 상태.
- TaskManager: `_move_waypoints_by_task` 등 캐시가 빈 상태.

반면 DB의 task는 그대로 남아, 재시작 직후 다음 두 가지가 동시에 깨진다.

1. **충돌 안전(occupancy)**: RUNNING 로봇이 점유/통과 중인 zone을 TrafficManager가 모름 → 새로 배차되는
   로봇이 그 zone을 피하지 못해 **충돌 위험**.
2. **task 수명주기**: RUNNING task의 **ROS Action goal handle이 소실**됨. ROS2 액션은 새 프로세스에서
   기존 goal에 재접속이 불가능하다 → `handle_task_result` 콜백이 영영 안 와서 그 task는 **영원히 멈춤**.

> 2번은 회피 불가능한 제약이다. in-flight 액션 task는 결국 재발행하거나, 리셋하거나, 다른 신호로 완료를
> 재동기해야 한다. "예약만 정확히 복원하면 그대로 이어진다"는 성립하지 않는다.

---

## 2. 모든 판단을 바꾸는 두 가지 사실

1. **로봇은 source로 되돌아가지 않는다.** `MoveCommand` goal에는 목적지(target)만 실려 가고
   (`_move_command_waypoints_for_task`), 로봇 Nav2는 **현재 위치에서 target으로** 다시 계획한다.
   즉 "source 복귀"는 물리적으로 일어나지 않는다.
2. **재시작해도 로봇은 원래 goal을 계속 수행 중이다.** action server는 로봇(State Manager) 프로세스에
   있고, Fleet Manager가 죽어도 살아서 목적지로 계속 주행한다. Fleet은 **결과 콜백만 잃은** 것이지
   로봇이 멈춘 게 아니다.

또한 **S1에서 텔레메트리(amcl_pose / picky_state)를 RobotStateMonitor가 받아 DB에 반영**하게 되었으므로,
복구 시 로봇의 **현재 위치/상태**를 알 수 있다. 이게 깔끔한 복구의 열쇠다.

---

## 3. 검토한 안

### A. (철회) DB `source_zone` 기준으로 재-BFS 후 재배차

RUNNING을 ASSIGNED로 리셋하고 기존 `_dispatch_ready_tasks`로 재배차 → BFS가 새 경로를 예약.

**문제(검토 중 제기되어 철회):** 재배차는 task의 **DB `source_zone`** 기준으로 BFS를 돌리는데, 로봇은
이미 source를 떠나 **경로 중간**에 있다. 경합 상황에서 재-BFS가 **로봇이 실제로 달리는 복도와 다른 복도**를
예약할 수 있다 → TrafficManager가 믿는 경로 ≠ 로봇의 실제 경로 → 다른 로봇을 그 실제 복도로 보내면 충돌.
점유 복원이 근사도 아니고 **틀린 곳**을 잡을 수 있다는 게 치명적. → **채택 안 함.**

### A′. 현재 위치 기준 재예약 + goal 재발행

로봇 **현재 zone → target**으로 재예약하고 `MoveCommand`를 다시 보낸다. 점유는 정확해진다.
단 로봇이 아직 옛 goal을 수행 중이므로, State Manager가 **새 goal을 받으면 기존 것을 취소·대체(preempt)**
해야 한다(안 그러면 이동 콜백 2중 실행). 재주행은 사실상 없음(같은 target).

### A″. 현재 위치 기준 재예약 + 텔레메트리 완료 재동기 (로봇 미개입) — **권장**

로봇은 어차피 target으로 잘 가고 있으니 **건드리지 않는다.** Fleet은:
- (점유) 로봇 **현재 zone → target**으로 재예약해 충돌만 막는다(기존 `reserve_path` 재사용).
- (완료) 결과 콜백 대신 **picky_state 전이**로 완료를 재동기한다. 로봇이 도착 상태
  (MOVING_* → WAITING_FOR_COBOT / STANDBY 등)로 바뀌면 그 MOVE를 SUCCESS 처리하고 flow 진행.
- (실패 보정) 일정 시간 진행/도착 신호가 없으면(로봇도 죽었거나 실패) 타임아웃 → FAILED + 재계획.

장점: **재주행 0, goal 재발행 없음(preempt 계약 불필요), 점유 정확.** S1 텔레메트리 덕분에 가능.

### B. 예약 경로를 DB에 영속화

`Task.reserved_path` 컬럼을 두고 예약 시 저장 → 재시작 때 정확 복원. 단 **스키마 변경(회색지대)** +
이미 지나온 노드까지 막는 과점유 + goal 재발행 문제는 그대로. 2대 규모에선 이득 대비 비용이 큼.

### 비교

| 안 | 점유 정확도 | 로봇 개입 | 시간 손실 | 스키마 | 비고 |
|----|------------|----------|----------|--------|------|
| A (source 기준) | 실제 경로와 어긋남 | 재발행 | 재주행 | 없음 | 충돌 위험 잔존 — 철회 |
| A′ (현재 위치+재발행) | 정확 | 재발행(preempt 계약 필요) | 거의 없음 | 없음 | 차선 |
| **A″ (현재 위치+텔레메트리 재동기)** | 정확 | **없음** | **0** | 없음 | **권장** |
| B (경로 영속화) | 정확(과점유) | 재발행 | - | 있음 | 무겁고 이득 작음 |

## 4. 권장안(A″) 컴포넌트별 책임

### FleetRepository (박서우)
- `list_recovery_tasks()` — RUNNING task + `robot_name, task_type, target_zone_name, order_id/stocking_item_id, sequence_no`, 그리고 로봇 `pos_x/pos_y/picky_state`(현재 위치 산출용)를 한 번에 조회.
- `fail_task_for_recovery(task_id)` — 타임아웃 fallback 전용(FAILED + 예약 해제 트리거).
- `has_emergency_robots()` — emergency 게이트 복원 판단.
- (A″는 RUNNING을 유지하므로 RUNNING→ASSIGNED 리셋은 기본 경로에선 불필요)

### TrafficManager (박서우)
- `nearest_zone(x, y) -> zone_name` — `_zone_coords` 최근접 노드(현재 위치 → 그래프 노드).
- 점유 재예약은 **기존 `reserve_path(robot, task_id, source=current_zone, target)` 재사용**(새 메서드 최소화).
- `rebuild_dock(robot_id, dock_name)` — CHARGING 로봇 도크 재점유(다른 로봇이 같은 도크 못 고르게).

### TaskManager (이명제)
- `reconcile_on_startup()` — executor spin 후 **~2초 one-shot**으로 1회(텔레메트리 도착·action server 탐색 보장 위해 `__init__` 동기 호출 금지). `_scheduler_lock` 보호.
  1. emergency 게이트 복원: `has_emergency_robots()`면 `_fleet_paused=True` + dispatch 금지 후 종료.
  2. RUNNING MOVE/DOCK: 로봇 현재 zone 산출 → `traffic.reserve_path(robot, task_id, current_zone, target)`로 점유 복원, `_move_waypoints_by_task` 채움. **로봇엔 명령 안 보냄.** task를 `_recovering`(deadline 포함)에 등록.
  3. CHARGE → `rebuild_dock`. COBOT → 텔레메트리/타임아웃 처리 대상으로만 등록.
  4. 점유 복원 후 정상 poll/dispatch 허용.
- 완료 재동기: `check_waiting_work`(5s poll)에서 `_recovering` task를 로봇 picky_state/위치와 대조 →
  도착이면 SUCCESS+advance, deadline 초과+미진행이면 `fail_task_for_recovery`+재계획.

### State Manager (박서우)
- A″에서는 재발행이 없어 **goal preempt 계약 불필요**(A′ 대비 장점). "도착 시 picky_state를 명확한 도착
  상태로 전이"하는 기존 계약만 신뢰하면 됨.

---

## 5. reconcile 실행 순서 (요약)

```
one-shot timer(~2s, spin 이후) -> TaskManager.reconcile_on_startup():
  with _scheduler_lock:
    tasks = repo.list_recovery_tasks()
    1) repo.has_emergency_robots() == True 면: _fleet_paused=True; (선택)e-stop 재전파; return
    2) CHARGE RUNNING 로봇 -> traffic.rebuild_dock(robot, dock)
    3) RUNNING MOVE/DOCK -> current = traffic.nearest_zone(pos); traffic.reserve_path(robot, task_id, current, target)
                            _recovering[task_id] = now + timeout
    4) _dispatch_ready_tasks()  # 점유가 복원됐으므로 신규 배차 충돌-안전
  이후 5s poll: _recovering 항목을 picky_state/위치로 완료 또는 타임아웃 처리
```

---

## 6. 엣지 케이스
- **EMERGENCY 중 재시작**: 게이트 복원이 최우선(안 막으면 e-stop인데 배차됨).
- **블랙아웃 중 이미 도착**: picky_state 도착 상태 → 즉시 SUCCESS 추론.
- **중간에 실패/정지**: 진행 신호 없음 → 타임아웃 → FAILED + 재계획.
- **COBOT RUNNING**: 현재 `send_cobot_task` 미구현이라 텔레메트리/타임아웃 처리 대상([S2]와 연동).
- **재예약 경합**: 2대·sparse 그래프라 드묾. 현재 위치 기준 BFS가 서로 막히면 한쪽은 hold(충돌보다 정지 선택).
- **reconcile 중복 실행 금지**: one-shot + lock.

## 7. 담당 경계 (이 작업 한정)
- 박서우: `nearest_zone`, `reserve_path`(현재 위치 인자) 재사용/검증, `rebuild_dock`, `list_recovery_tasks`/`fail_task_for_recovery`/`has_emergency_robots`, State Manager 도착 상태 전이 신뢰성.
- 이명제: `reconcile_on_startup` 오케스트레이션 + 완료 재동기/타임아웃 로직 + one-shot timer 배선.
- 회색지대: B로 갈 경우의 스키마(`Task.reserved_path`).

---

## 8. 함께 정할 것 (회의 안건)

1. **A″ vs A′**: 로봇 미개입+텔레메트리 재동기(A″, 권장) vs 현재 위치 재예약+goal 재발행(A′, State Manager preempt 계약 필요). 둘 다 "현재 위치 기준"이라 점유는 정확. 차이는 "로봇을 건드리느냐".
2. **완료 재동기 방식**: 5s poll에서 대조(간단, 추가 배선 없음) vs picky_state 이벤트를 TaskManager로 직접 라우팅(즉시성↑, 배선 추가).
3. **타임아웃 값**: 도착 신호 대기 한계(예: MOVE 기대 소요 + 여유). 초과 시 FAILED+재계획.
4. **emergency 재전파 여부**: 재시작이 EMERGENCY 상태였을 때 로봇에 e-stop을 다시 보낼지, DB 게이트만 유지할지.
5. **COBOT 처리**: ExecuteTask 미구현 상태에서 RUNNING COBOT을 어떻게(유지/타임아웃/리셋).
6. **B(경로 영속화) 도입 여부**: 재주행 절대 불가 요구가 생기면 재검토. 현재는 보류 권장.
