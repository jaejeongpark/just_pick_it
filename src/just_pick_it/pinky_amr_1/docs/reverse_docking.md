# Reverse Docking System

## 개요

자율주행 로봇 **Pinky**가 Standby Zone에 도착한 후, ArUco 마커를 기준으로 충전 도크에 정밀 후진 주차하는 시스템이다.

AMCL과 Odometry만으로는 도킹에 필요한 수 센티미터 수준의 정밀도를 확보할 수 없다는 문제에서 출발하였다. 이를 해결하기 위해 ArUco 마커를 절대 위치 기준점으로 삼는 **시각 서보(Visual Servoing)** 방식을 채택하였다.

---

## 시스템 환경

| 항목 | 사양 |
|------|------|
| 로봇 | Pinky (교육용 자율주행 로봇) |
| 카메라 | Raspberry Pi Camera Module 3 Wide (Sony IMX708, 최대 120° FOV) |
| 위치 추정 | AMCL + LiDAR |
| 구동 방식 | Differential Drive (`cmd_vel`) |
| 프레임워크 | ROS 2 |
| 비전 라이브러리 | OpenCV ArUco |

---

## 도킹 시나리오

```
[Standby Zone]
       │  Nav2로 이동 (AMCL 기반)
       ▼
[Standby Zone 도착]
       │  Nav2 종료, Docking Node 인계
       ▼
[Phase 0]  마커 탐색 및 yaw 정렬
       ▼
[Phase 1]  횡방향 pre-alignment
       ▼
[Phase 2]  ArUco 시각 서보 후진
       ▼
[Phase 3]  주차라인 추종 + 정지선 감지
       ▼
[도킹 완료]  AMCL 위치 재초기화
```

---

## 단계별 설계

---

### Phase 0 — 마커 탐색 및 Yaw 정렬

**목적**: 로봇이 ArUco 마커를 정면으로 바라보도록 제자리 회전 정렬한다.

**동작**
- 마커가 검출되지 않으면 저속으로 탐색 회전
- 마커 검출 시 `rvec[1]` (카메라 기준 마커 yaw) 오차를 PID로 수렴

```
angular.z = PID(-rvec[1])
linear.x  = 0.0
종료 조건: |rvec[1]| ≤ 0.05 rad
```

**고민한 지점**

픽셀 기반으로 마커의 화면 중앙 정렬(cx_err)을 목표로 삼는 방식을 먼저 고려했다. 그러나 픽셀 중앙에 마커가 오더라도 로봇 자체의 yaw 오차가 남아 있을 수 있다는 문제가 있었다.

**선택 이유**

`rvec[1]`은 카메라와 마커 사이의 실제 yaw 각도를 미터 공간에서 표현한 값이므로, 픽셀 좌표보다 물리적으로 정확한 정렬 기준이 된다. 이후 Phase 2의 PID 제어와도 동일한 좌표계를 공유하여 일관성을 유지한다.

**적용 기술**: `cv2.solvePnP` + `rvec[1]` 기반 PID

---

### Phase 1 — 횡방향 Pre-alignment

**목적**: 후진 시작 전 `tvec[0]` (카메라 기준 마커의 좌우 오프셋)을 허용 범위 이내로 줄여 PID 수렴 속도를 높인다.

**동작**
- `tvec[0]` 오차에 비례하는 각속도로 전진 아크(arc) 이동
- 마커가 우측(`tvec[0] > 0`)이면 좌회전하며 전진

```
angular.z = -Kp * tvec[0]
linear.x  =  0.04 m/s
종료 조건: |tvec[0]| ≤ 0.03 m
```

**고민한 지점**

Differential Drive 로봇은 제자리 횡이동이 불가능하다. 제자리 회전만으로 `tvec[0]`을 줄이는 방법도 검토했지만, yaw 회전은 `tvec[0]`보다 `rvec[1]`에 직접 영향을 주므로 효과가 제한적이다.

또한 Phase 2의 PID만으로 초기 횡방향 오차를 전부 보정하면 도킹 경로가 굽어져 최종 정렬 정밀도가 낮아질 수 있다고 판단했다.

**선택 이유**

전진 아크(회전 + 전진 동시) 방식이 Differential Drive에서 횡방향 위치를 보정하는 가장 자연스러운 수단이다. Phase 2 시작 전 초기 오차를 3cm 이내로 줄여 수렴 시간을 단축한다.

**적용 기술**: `tvec[0]` 비례 제어 + 전진 아크

---

### Phase 2 — ArUco 시각 서보 후진

**목적**: ArUco `tvec`과 `rvec`를 실시간으로 읽어 PID로 각도를 보정하며 후진한다.

**동작**
- `tvec[0]`: 횡방향 오차 → angular.z 보정
- `rvec[1]`: yaw 오차 → angular.z 보정
- `tvec[2]`: 마커까지 거리 → 감속 프로파일 및 Phase 3 전환 기준
- 마커 미검출 시 각도 보정 없이 저속 직진 후진 유지

```
# 후진 시 보정 방향이 전진과 반대 (부호 반전)
angular.z = -(lat_pid(tvec[0]) + yaw_pid(rvec[1]))
linear.x  = -min(v_max, tvec[2] * 0.15 + 0.02)
종료 조건: tvec[2] ≤ 0.20 m
```

**고민한 지점**

**① 제어 기준: Pixel-space vs Metric-space**

초기 설계에서는 마커의 화면 중앙 편차(`cx_err`, 픽셀)로 각도를 보정하는 방식을 사용했다. 그러나 픽셀 오차는 렌즈 왜곡과 원근 왜곡의 영향을 받으며, 거리가 달라질 때 동일 픽셀 편차가 다른 물리적 오차를 의미하게 된다.

`tvec[0]`과 `rvec[1]`은 카메라 행렬과 왜곡 계수를 반영한 `solvePnP` 결과로, 실제 미터 공간의 오차를 표현한다. 거리와 무관하게 일관된 PID 게인을 사용할 수 있고, Wide lens의 강한 왜곡 환경에서도 안정적이다.

**② 후진 시 제어 부호 반전**

전진 주행에서 마커가 우측(`tvec[0] > 0`)이면 우회전(angular.z > 0)으로 마커를 중앙으로 가져온다. 그러나 후진 주행에서는 동일 상황에 좌회전(angular.z < 0)이 필요하다. 후진 시 회전 방향이 로봇 후미 이동 방향에 반영되기 때문이다. 이 부호 반전을 놓치면 제어가 발산한다.

**③ 위치 추정 소스 선택**

도킹 구간에서 AMCL과 Odometry 중 어느 것을 기준으로 삼을지 검토했다.

| | AMCL | Odometry | ArUco |
|---|---|---|---|
| 정밀도 | ~10cm | 누적 오차 | ~1cm |
| 갱신 주기 | LiDAR 스캔 주기 | 바퀴 인코더 주기 | 카메라 프레임 주기 |
| 절대 기준 | 맵 기반 | 상대 기준 | 마커 기반 절대 |

도킹에 필요한 정밀도(~1cm)는 ArUco만이 제공할 수 있다. AMCL은 Standby Zone까지의 내비게이션에만 사용하고, 도킹 구간은 ArUco 시각 서보로 완전히 대체하는 방식을 선택했다.

**적용 기술**: `cv2.solvePnP` + `tvec/rvec` dual PID + 거리 기반 감속

---

### Phase 3 — 주차라인 추종 및 정지선 감지

**목적**: ArUco 검출 한계 거리(20cm) 이내의 최종 구간을 노란 주차라인으로 중심을 맞추며 후진하고, 파란 정지선을 감지하면 정지한다.

**동작**
- 프레임 하단 2/3 ROI에서 노란 HSV 마스크로 좌/우 라인 중심 추출
- 두 중심의 평균과 프레임 중앙의 편차를 비례 제어
- 파란 HSV 마스크에서 한 row의 가로 커버리지가 50% 이상이면 정지

```
# 주차라인 중심 추종
lane_err  = (left_cx + right_cx) / 2 - frame_cx
angular.z = -Kp_lane * lane_err
linear.x  = -0.03 m/s

# 파란 정지선 감지
row_coverage = sum(blue_mask, axis=1) / frame_width
정지 조건: any(row_coverage > 0.5)
```

**고민한 지점**

**① 도킹 완료 감지 방법**

물리 접촉 센서와 충전 신호가 없는 환경에서 세 가지 방법을 비교했다.

| 방법 | 장점 | 단점 |
|------|------|------|
| ArUco `tvec[2]` 임계값 | 추가 설치 불필요 | Phase 3 진입 후 마커 재검출 어려움 |
| 주차라인 끝점 픽셀 위치 | 추가 설치 불필요 | 끝점 검출이 복잡하고 불안정 |
| 마킹테이프 가로선 | 명확한 단일 이벤트 감지 | 테이프 추가 설치 필요 |

주차라인 끝점 방식은 라인이 완전히 시야에서 사라지는 순간을 정확히 잡기 어렵고, 조명 변화에 민감했다. 가로 마킹테이프는 "특정 row의 가로 커버리지"라는 단순한 조건으로 정지 이벤트를 명확히 정의할 수 있어 채택했다.

**② 테이프 색상 선택**

바닥(검정), 주차라인(노랑), 마커 벽(흰색)과 모두 대비되는 색이 필요했다. 빨강도 대비 조건을 만족하지만, HSV에서 Hue가 0°와 180° 양쪽에 걸쳐 마스크를 두 번 생성해야 하는 구현 복잡도가 있었다. 파랑은 HSV 범위가 단일 구간(H: 100–130)으로 구현이 단순하고, 충전 구역처럼 안전/중립 이미지를 줄 수 있어 선택했다.

**③ 테이프와 주차라인의 배치**

테이프를 주차라인 안쪽에만 붙이는 방식과 주차라인 위를 포함한 전체 폭으로 붙이는 방식을 검토했다. 안쪽 배치는 프레임 전체 폭 대비 커버리지 비율이 낮아 임계값 조정이 필요하고 불안정했다. 파란 테이프를 노란 테이프 위에 겹쳐 붙이면 파란색이 표면을 덮기 때문에 색상 혼동 없이 전체 폭 커버리지를 확보할 수 있어 겹침 방식을 선택했다.

**④ 카메라 렌즈 선택**

Standard 렌즈(수평 66°)와 Wide 렌즈(수평 102°)를 비교했다.

| | Standard | Wide |
|---|---|---|
| Standby Zone에서 마커 시야 | ±33° 이내 필요 | ±51° 이내면 충분 |
| ArUco 검출 정밀도 | 높음 | 왜곡 보정 필요 |
| 가까운 거리 왜곡 | 낮음 | 높음 |

도킹 초기에 로봇이 정확한 각도로 Standby Zone에 도착하지 못할 수 있기 때문에, 마커 탐색 회전 범위의 여유가 더 중요했다. Wide 렌즈는 왜곡이 강한 대신 `solvePnP`에 왜곡 계수를 적용하면 보정되므로, 카메라 캘리브레이션을 전제로 Wide 렌즈를 선택했다.

**적용 기술**: HSV 마스크 기반 노란 주차라인 중심 추종 + 파란 마킹테이프 row coverage 감지

---

### 도킹 완료 — AMCL 위치 재초기화

**목적**: 도킹 완료 후 충전 도크의 알려진 절대 좌표로 AMCL을 재초기화하여 이후 내비게이션 기준점을 정확하게 한다.

**동작**

도킹이 완료된 시점에 로봇의 위치는 충전 도크의 설계 좌표와 같다. 이 값을 `/initialpose` 토픽으로 발행하면 AMCL 파티클 필터가 해당 좌표 주변으로 수렴한다.

```python
# Odometry나 TF를 읽는 것이 아닌, 알려진 dock 절대 좌표를 직접 발행
publish /initialpose(dock_map_x, dock_map_y, dock_map_yaw)
```

**선택 이유**

도킹 전까지 쌓인 AMCL 오차를 도킹 완료 시점에 한 번에 보정한다. 도킹 완료 위치는 설계상 확정된 절대 좌표이므로, TF로 현재 위치를 읽어 발행하는 것보다 이 값을 직접 사용하는 것이 정확하다.

## 파라미터 구조

캘리브레이션이 필요한 항목과 튜닝이 필요한 항목을 분리하여 관리한다.

```yaml
# 비전 담당 캘리브레이션 후 실측값 입력
camera_matrix: [fx, 0, cx, 0, fy, cy, 0, 0, 1]
dist_coeffs:   [k1, k2, p1, p2, k3]

# 환경 설치 후 실측 튜닝
dock_switch_distance: 0.20    # 마커 불안정 진입 거리
tape_coverage_thresh: 0.5     # 정지선 감도
lane_kp: 0.003                # 주차라인 반응 속도
```
