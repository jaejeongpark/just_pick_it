# Reverse Docking System

## 개요

자율주행 로봇 **Pinky**가 Standby Zone에 도착한 후, ArUco 마커를 기준으로 충전 도크에 정밀 후진 주차하는 시스템이다.

AMCL과 Odometry만으로는 도킹에 필요한 수 센티미터 수준의 정밀도를 확보할 수 없다는 문제에서 출발하였다. 이를 해결하기 위해 ArUco 마커를 절대 위치 기준점으로 삼는 **시각 서보(Visual Servoing)** 방식을 채택하였다.

---

## 시스템 환경

| 항목 | 사양 |
|------|------|
| 로봇 | Pinky (교육용 자율주행 로봇) |
| 카메라 | Raspberry Pi Camera Module 3 Wide (Sony IMX708, 최대 120° FOV) |
| 위치 추정 | AMCL + LiDAR |
| 구동 방식 | Differential Drive (`cmd_vel`) |
| 프레임워크 | ROS 2 |
| 비전 라이브러리 | OpenCV ArUco (`solvePnP`) |
| 도킹 환경 | 바닥: 검정 / 주차라인: 노랑 / 마커 벽: 흰색 |

---

## 전체 흐름

```
[Standby Zone]
       │  Nav2로 이동 (AMCL 기반)
       ▼
[Standby Zone 도착]
       │  Nav2 종료 → Docking Node 인계
       ▼
[Phase 0]  마커 탐색 및 Yaw 정렬
       ▼
[Phase 1]  횡방향 Pre-alignment
       ▼
[Phase 2]  ArUco 시각 서보 후진
       ▼
[Phase 3]  주차라인 추종 + 정지선 감지
       ▼
[도킹 완료]  AMCL 위치 재초기화
```

---

## 단계별 설계

---

### Phase 0 — 마커 탐색 및 Yaw 정렬

**목적**: 로봇이 ArUco 마커를 정면으로 바라보도록 제자리 회전 정렬한다.

#### 기술 선택

| 방법 | 설명 | 단점 | 선택 |
|------|------|------|:----:|
| 픽셀 중앙 정렬 (`cx_err`) | 마커가 화면 중앙에 오도록 회전 | 픽셀 중앙 ≠ 실제 yaw 정렬, 렌즈 왜곡 영향 | |
| `rvec[1]` 기반 PID | 카메라-마커 간 실제 yaw 각도로 제어 | 캘리브레이션 필요 | ✅ |

`rvec[1]`은 `solvePnP`가 반환하는 미터 공간의 yaw 각도로, 픽셀 좌표보다 물리적으로 정확하다. Phase 2의 PID와 동일한 좌표계를 공유하여 일관성을 유지한다.

#### 제어

```
마커 미검출  →  저속 탐색 회전 (angular.z = 0.3)
마커 검출    →  angular.z = PID(-rvec[1]),  linear.x = 0
종료 조건   :  |rvec[1]| ≤ 0.05 rad
```

---

### Phase 1 — 횡방향 Pre-alignment

**목적**: 후진 시작 전 `tvec[0]` (마커 기준 좌우 오프셋)을 허용 범위 이내로 줄여 Phase 2 PID의 수렴 속도를 높인다.

#### 기술 선택

Differential Drive 로봇은 제자리 횡이동이 불가능하므로 회전과 이동을 조합한 아크(arc) 기동이 필요하다.

| 방법 | 설명 | 단점 | 선택 |
|------|------|------|:----:|
| 제자리 회전 | yaw만 바뀌고 tvec[0]은 거의 개선되지 않음 | 횡방향 보정 효과 없음 | |
| 전진 아크 | 회전하며 전진 | 마커 벽 방향으로 접근 → 충돌 위험 | |
| 후진 아크 | 회전하며 후진 | 벽에서 멀어지며 도크 방향으로 이동 | ✅ (기본) |
| 전진 아크 (fallback) | 후진 중 너무 가까워졌을 때 전환 | 전진이지만 보정 완료까지만 사용 | ✅ (보조) |

사람이 주차할 때처럼 후진을 기본으로 하되, `tvec[2]`가 임계값에 도달하면 전진 아크로 전환하여 벽 충돌 없이 보정을 완료한다.

#### 제어

```
tvec[2] > dock_switch_distance  →  후진 아크 (기본)
    linear.x  = -0.04 m/s
    angular.z = -Kp * tvec[0]     # 후진 부호 반전

tvec[2] ≤ dock_switch_distance  →  전진 아크 (fallback)
    linear.x  = +0.04 m/s
    angular.z = +Kp * tvec[0]     # 전진 부호 정방향

종료 조건: |tvec[0]| ≤ 0.03 m
```

> **부호 반전**: 후진 시 회전 방향이 로봇 후미 이동 방향에 반영되기 때문에 전진과 반대 부호를 사용한다.

---

### Phase 2 — ArUco 시각 서보 후진

**목적**: ArUco `tvec`과 `rvec`를 실시간으로 읽어 PID로 각도를 보정하며 도크 방향으로 후진한다.

#### 기술 선택 — 위치 추정 소스

| 소스 | 정밀도 | 갱신 주기 | 절대 기준 | 선택 |
|------|--------|-----------|-----------|:----:|
| AMCL | ~10 cm | LiDAR 스캔 | 맵 기반 | 내비게이션 전용 |
| Odometry | 누적 오차 | 인코더 | 상대 기준 | |
| ArUco `tvec/rvec` | ~1 cm | 카메라 프레임 | 마커 기반 절대 | ✅ |

도킹 구간은 AMCL/Odometry를 완전히 배제하고 ArUco 시각 서보로 대체한다.

#### 기술 선택 — 제어 공간

| 제어 기준 | 특징 | 단점 | 선택 |
|-----------|------|------|:----:|
| Pixel-space (`cx_err`) | 구현 단순 | 렌즈 왜곡, 원근 왜곡 영향 / 거리마다 게인 재조정 필요 | |
| Metric-space (`tvec/rvec`) | 실제 미터 단위 오차 / 거리 무관 일관된 게인 | 캘리브레이션 필요 | ✅ |

#### 제어

```
angular.z = -(lat_pid(tvec[0]) + yaw_pid(rvec[1]))   # 후진 부호 반전
linear.x  = -min(v_max, tvec[2] * 0.15 + 0.02)       # 거리 비례 감속

마커 미검출 시  →  각도 보정 없이 저속 직진 후진 유지
종료 조건      :  tvec[2] ≤ 0.20 m  →  Phase 3 전환
```

---

### Phase 3 — 주차라인 추종 및 정지선 감지

**목적**: ArUco 검출 한계 거리 이내의 최종 구간을 노란 주차라인으로 중심을 맞추며 후진하고, 파란 정지선을 감지하면 정지한다.

#### 기술 선택 — 도킹 완료 감지

물리 접촉 센서와 충전 신호가 없는 환경에서 세 가지 방법을 비교했다.

| 방법 | 장점 | 단점 | 선택 |
|------|------|------|:----:|
| ArUco `tvec[2]` 임계값 | 추가 설치 불필요 | Phase 3 진입 후 마커 재검출 어려움 | |
| 주차라인 끝점 픽셀 위치 | 추가 설치 불필요 | 끝점 검출 복잡, 조명 변화에 민감 | |
| 파란 마킹테이프 가로선 | 단일 이벤트로 명확히 정의 가능 | 테이프 추가 설치 필요 | ✅ |

#### 기술 선택 — 테이프 색상

| 색상 | 바닥(검정) 대비 | 주차라인(노랑) 대비 | 마커 벽(흰색) 대비 | HSV 구현 | 선택 |
|------|:--------------:|:-----------------:|:-----------------:|---------|:----:|
| 노랑 | ✅ | ❌ 주차라인과 동일 | ✅ | 단순 | |
| 빨강 | ✅ | ✅ | ✅ | H가 0°/180° 양쪽에 걸쳐 마스크 2회 필요 | |
| 파랑 | ✅ | ✅ | ✅ | H: 100–130 단일 구간 | ✅ |

#### 기술 선택 — 테이프 배치

| 배치 | 설명 | 단점 | 선택 |
|------|------|------|:----:|
| 주차라인 안쪽 | 양 라인 사이에만 붙임 | 전체 프레임 폭 대비 커버리지 낮아 임계값 불안정 | |
| 주차라인 전체 폭 겹침 | 노란 테이프 위에 파란 테이프 겹침 | 없음 (파란색이 표면 덮어 혼동 없음) | ✅ |

#### 제어

```
# 횡방향 제어 (노란 주차라인 HSV 마스크)
lane_err  = (left_cx + right_cx) / 2 - frame_cx
angular.z = -Kp_lane * lane_err
linear.x  = -0.03 m/s

# 정지 조건 (파란 마킹테이프 HSV 마스크)
row_coverage = sum(blue_mask > 0, axis=1) / frame_width
정지: any(row_coverage > 0.5)     # 가로 50% 이상 덮이면 정지
```

---

### 도킹 완료 — AMCL 위치 재초기화

**목적**: 도킹 완료 후 충전 도크의 알려진 절대 좌표로 AMCL을 재초기화하여 이후 내비게이션의 기준점을 정확하게 한다.

| 방법 | 설명 | 단점 | 선택 |
|------|------|------|:----:|
| TF로 현재 위치 읽어 발행 | 추가 좌표 불필요 | 도킹 과정의 오차가 그대로 반영됨 | |
| 알려진 dock 절대 좌표 직접 발행 | 설계상 확정된 정확한 위치 사용 | dock 좌표를 사전에 측정해야 함 | ✅ |

```python
# dock 절대 좌표를 /initialpose 로 발행 → AMCL 파티클 필터 재수렴
publish /initialpose(dock_map_x, dock_map_y, dock_map_yaw)
```

## 파라미터 구조

| 분류 | 파라미터 | 설명 |
|------|---------|------|
| **캘리브레이션** (비전 담당 입력) | `camera_matrix` | Wide lens 내부 파라미터 |
| | `dist_coeffs` | 왜곡 계수 `[k1, k2, p1, p2, k3]` |
| **ArUco** | `marker_size_m` | 마커 한 변 실측 길이 (m) |
| | `dock_switch_distance` | Phase 2→3 전환 거리 / Phase 1 전진 전환 기준 (m) |
| **Phase 0** | `rotate_kp/ki/kd` | Yaw 정렬 PID 게인 |
| | `rotate_thresh_rad` | 정렬 허용 오차 (rad) |
| **Phase 1** | `prealign_kp` | 횡방향 비례 게인 |
| | `prealign_thresh_m` | 횡방향 허용 오차 (m) |
| **Phase 2** | `lat_kp/ki/kd` | 횡방향 PID 게인 |
| | `yaw_kp/ki/kd` | Yaw PID 게인 |
| | `reverse_speed` | 최대 후진 속도 (m/s) |
| **Phase 3** | `lane_kp` | 주차라인 비례 게인 (pixel 기반) |
| | `final_speed` | 최종 후진 속도 (m/s) |
| | `tape_hsv_lower/upper` | 파란 정지선 HSV 범위 |
| | `tape_coverage_thresh` | 정지선 감지 커버리지 임계값 |
