# autonomous_sys_ws 협업 인수인계서

## 이 문서를 읽기 전에 꼭 확인할 것

이 워크스페이스는 **두 종류의 git 저장소**로 구성되어 있습니다.
수정한 패키지가 어느 저장소에 속하느냐에 따라 git 절차가 달라지므로, 아래 구조를 먼저 이해하세요.

```
autonomous_sys_ws/         ← 저장소 ①  (github.com/jaejeongpark/autonomous_sys_ws)
├── src/
│   ├── pinky_pro/         ← 저장소 ②  (github.com/jaejeongpark/pinky_pro)
│   │   ├── pinky_bringup/
│   │   ├── pinky_gz_sim/
│   │   ├── pinky_navigation/
│   │   └── ...
│   ├── mycobot_ros2/      ← 저장소 ③  (github.com/jaejeongpark/mycobot_ros2)
│   └── <프로젝트 패키지>/ ← 저장소 ① 에 포함
├── scripts/
└── README.md
```

| 수정한 파일 위치 | 해당하는 저장소 | 따를 절차 |
|-----------------|----------------|----------|
| `src/pinky_pro/` 안 | 저장소 ② (pinky_pro) | **[패턴 A]** |
| `src/mycobot_ros2/` 안 | 저장소 ③ (mycobot_ros2) | **[패턴 A]** |
| `src/<프로젝트 패키지>/` 안 | 저장소 ① (autonomous_sys_ws) | **[패턴 B]** |

---

## 1. 최초 1회 설정

처음 이 워크스페이스를 받을 때만 실행합니다.

```bash
# 1. 워크스페이스 클론 (서브모듈까지 함께 받기)
git clone --recurse-submodules https://github.com/jaejeongpark/autonomous_sys_ws.git

# 2. 워크스페이스 폴더로 이동
cd autonomous_sys_ws

# 3. 서브모듈을 각각의 작업 브랜치로 전환
cd src/pinky_pro
git checkout my-features
cd ../mycobot_ros2
git checkout my_feature
cd ../..

# 4. 빌드
colcon build --symlink-install

# 5. ROS2 환경 설정 (매 터미널 세션마다 필요)
source install/setup.bash
```

> **확인 방법:** `ls src/pinky_pro/` 를 실행했을 때 `pinky_bringup`, `pinky_gz_sim` 등 패키지 폴더가 보이고, `ls src/mycobot_ros2/` 를 실행했을 때 패키지 폴더가 보이면 정상입니다.

---

## 2. 매일 작업 시작 전 (최신 코드 받기)

다른 팀원이 올린 변경사항을 받아야 합니다. **코드 수정 전에 항상 실행**하세요.

```bash
# 1. 워크스페이스 루트로 이동
cd ~/autonomous_sys_ws

# 2. 워크스페이스 최신화 (저장소 ①)
git pull origin main

# 3. pinky_pro 등 서브모듈 최신화 (저장소 ②)
git submodule update --remote

# 4. 새로 받은 코드 빌드
colcon build --symlink-install

# 5. 환경 설정
source install/setup.bash
```

---

## 패턴 A: pinky_pro 또는 mycobot_ros2 안의 패키지를 수정했을 때

`src/pinky_pro/` 또는 `src/mycobot_ros2/` 아래에 있는 패키지를 수정했을 때의 절차입니다.

**총 2개의 저장소에 각각 push해야 합니다.**

> **브랜치 이름 주의:** 두 서브모듈의 작업 브랜치 이름이 다릅니다.
> - `pinky_pro` → `my-features`
> - `mycobot_ros2` → `my_feature`

### Step 1. 해당 서브모듈 폴더로 이동

```bash
# pinky_pro를 수정한 경우
cd ~/autonomous_sys_ws/src/pinky_pro

# mycobot_ros2를 수정한 경우
cd ~/autonomous_sys_ws/src/mycobot_ros2
```

### Step 2. 현재 브랜치 확인

```bash
git branch
```

- `pinky_pro`는 `* my-features`, `mycobot_ros2`는 `* my_feature` 가 보여야 합니다.

```
  main
* my-features      ← pinky_pro의 경우
```

브랜치가 다르면 아래 명령어로 전환하세요.

```bash
git checkout my-features    # pinky_pro
git checkout my_feature     # mycobot_ros2
```

### Step 3. 다른 사람 변경사항 먼저 받기

```bash
git pull origin my-features   # pinky_pro
git pull origin my_feature    # mycobot_ros2
```

### Step 4. 내가 수정한 파일 확인

```bash
git status
```

빨간색으로 표시된 파일들이 수정된 파일입니다.

### Step 5. 수정한 파일 스테이지에 올리기

```bash
# 방법 1: 특정 파일만 올리기 (권장)
git add pinky_gz_sim/launch/gazebo.launch.py

# 방법 2: 수정된 파일 전체 올리기
git add .
```

### Step 6. 커밋하기

```bash
git commit -m "fix: 변경 내용을 한 줄로 요약"
```

커밋 메시지 앞에는 아래 중 하나를 붙이세요.

| 접두사 | 사용 상황 |
|--------|----------|
| `feat:` | 새 기능 추가 |
| `fix:` | 버그 수정 |
| `chore:` | 설정·빌드 변경 |
| `docs:` | 문서 수정 |

예시: `git commit -m "fix: nav2 기본 맵 경로 수정"`

### Step 7. 해당 서브모듈 GitHub에 push

```bash
git push origin my-features   # pinky_pro
git push origin my_feature    # mycobot_ros2
```

### Step 8. 워크스페이스 루트로 돌아가기

```bash
cd ~/autonomous_sys_ws
```

### Step 9. 워크스페이스 저장소에도 업데이트 기록하기

```bash
# pinky_pro를 수정한 경우
git add src/pinky_pro
git commit -m "chore: update pinky_pro submodule"

# mycobot_ros2를 수정한 경우
git add src/mycobot_ros2
git commit -m "chore: update mycobot_ros2 submodule"

git push origin main
```

> **왜 Step 9이 필요한가요?**
> `autonomous_sys_ws` 저장소는 각 서브모듈이 어느 버전(커밋)을 가리키는지 기록합니다.
> Step 7만 하면 서브모듈 저장소에는 반영되지만, 워크스페이스 저장소의 기록은 이전 버전을 가리킨 채로 남습니다.
> 다른 팀원이 `git pull`로 받을 때 혼란이 없도록 Step 9까지 완료하세요.

## 패턴 B: 프로젝트 패키지를 수정했을 때

`src/pinky_pro/` 가 아닌, 직접 개발하는 프로젝트 패키지를 수정했을 때의 절차입니다.
저장소가 하나이므로 패턴 A보다 단순합니다.

### Step 1. 워크스페이스 루트에서 시작

```bash
cd ~/autonomous_sys_ws
```

### Step 2. 내가 수정한 파일 확인

```bash
git status
```

### Step 3. 수정한 파일 스테이지에 올리기

```bash
# 특정 패키지 폴더 전체
git add src/<패키지명>/

# 또는 특정 파일
git add src/<패키지명>/some_file.py
```

### Step 4. 커밋하기

```bash
git commit -m "feat: 변경 내용을 한 줄로 요약"
```

### Step 5. GitHub에 push

```bash
git push origin main
```

## 자주 발생하는 문제 해결

### `src/pinky_pro` 또는 `src/mycobot_ros2` 안이 비어 있어요

서브모듈 초기화가 안 된 상태입니다.

```bash
cd ~/autonomous_sys_ws
git submodule update --init --recursive
cd src/pinky_pro
git checkout my-features
cd ../mycobot_ros2
git checkout my_feature
cd ../..
```

---

### push했는데 팀원이 내 변경사항을 못 받아요

패턴 A에서 Step 7(서브모듈 push)만 하고 Step 9(워크스페이스 push)를 빠뜨린 경우입니다.

```bash
cd ~/autonomous_sys_ws
# 수정한 서브모듈에 맞게 선택
git add src/pinky_pro       # pinky_pro를 수정한 경우
git add src/mycobot_ros2    # mycobot_ros2를 수정한 경우
git commit -m "chore: update <서브모듈명> submodule"
git push origin main
```

---

### push 시 아래 에러가 나요

```
 ! [rejected]  my-features -> my-features (non-fast-forward)
error: failed to push some refs
```

다른 팀원이 먼저 push했습니다. pull을 먼저 하세요.

```bash
git pull origin my-features   # (pinky_pro 폴더 안에서)
git pull origin my_feature    # (mycobot_ros2 폴더 안에서)
# 또는
git pull origin main           # (autonomous_sys_ws 루트에서)
```

pull 후 다시 push하세요.

---

### `git branch` 결과에 `* (HEAD detached at ...)` 가 보여요

서브모듈이 특정 브랜치에 붙어 있지 않은 상태입니다.

```bash
cd ~/autonomous_sys_ws/src/pinky_pro
git checkout my-features

# mycobot_ros2인 경우
cd ~/autonomous_sys_ws/src/mycobot_ros2
git checkout my_feature
```

---

### 빌드 후 변경사항이 반영이 안 돼요

```bash
cd ~/autonomous_sys_ws
colcon build --symlink-install
source install/setup.bash
```

`source install/setup.bash`는 **새 터미널을 열 때마다** 실행해야 합니다.
매번 자동으로 실행하려면 아래를 `~/.bashrc`에 추가하세요.

```bash
source ~/autonomous_sys_ws/install/setup.bash
```
