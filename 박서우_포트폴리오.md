# 박서우 — Just Pick It 포트폴리오

> 자율주행 AMR(픽업 로봇) 2대 + 협동로봇(Cobot) 2대를 하나의 관제 시스템으로 묶어
> 음성/웹 주문을 받아 상품을 자동 픽업하는 ROS2 기반 멀티로봇 풀필먼트 프로젝트.

**담당 역할:** 멀티로봇 **상태·교통·데이터 계층**(Fleet Manager 3개 모듈) + **PICKY AMR 주행 인프라** 전담
**기여:** 79개 커밋 (전체 233개 중)

## 슬라이드 1 — 내 담당 모듈과 시스템 위치

```
  Web/음성 주문 ─HTTP/WS─▶  FleetManagerNode (🤝공동 조립)
                            ├ TaskManager        👤이명제
                            ├ RobotCommandGateway 👤이명제
                            ├ FleetApiServer      👤이명제
                            ├ TrafficManager      ✅박서우  구역 BFS·점유/도크 예약
                            ├ RobotStateMonitor   ✅박서우  텔레메트리 구독→DB
                            └ FleetRepository     ✅박서우  DB 단독 접근
                                      ▲                            │ ROS2 Action
              상태/배터리/위치 ◀──────┘                            ▼
                            State Manager(PICKY)  ✅박서우 ◀─ PICKY AMR(picky1/2)
```

### 내가 만든 핵심 (각 모듈의 역할)
- **TrafficManager** — 구역 그래프 BFS 경로 탐색 + **구역 단일 점유**·충전 도크 예약. 멀티로봇 충돌·교착 방지의 핵심.
- **RobotStateMonitor** — `picky_state/battery/amcl_pose` 토픽을 구독해 1Hz로 변경분만 DB 반영(coalesce). 텔레메트리 정합성 담당.
- **FleetRepository** — DB 단독 소유 계층(공용 `just_pick_it_db`로 분리). 데이터 진입점 단일화.
- **State Manager(PICKY)** — AMR 측 상태 머신. 배터리 STANDBY 전환·undock 분리·도착 사방향 정지회전.

## 슬라이드 2 — 핵심 문제 해결 (내 담당 범위 내 난제 3선)

### 1. 멀티로봇 네임스페이스 — "2대를 한 도메인에서 안 부딪히게" *(PICKY 주행 인프라)*
- **문제:** picky1/picky2가 같은 `ROS_DOMAIN_ID`에서 `/scan`·`/odom`·`/tf`·노드 이름까지 충돌.
- **원인:** tf2 라이브러리가 `/tf`를 **글로벌로 하드코딩** → 단순 namespace로는 분리 불가.
- **해결:** `PushROSNamespace`(노드·상대 토픽) + `SetRemap(/tf→/pickyN/tf)`(절대 토픽) 조합, `frame_prefix`는 비워 무접두어 프레임 유지. launch를 namespace 파라미터화해 picky2가 동일 코드 재사용.

### 2. TrafficManager — "좁은 아레나에서 충돌·교착 없이" *(내 전담 모듈)*
- **경로 예약:** 구역 BFS 탐색 + **구역 단일 점유**(한 구역 1대) 예약.
- **충전 도크 점유:** 복귀 로봇이 빈 도크로만 도킹. **콜드 스타트 시 양쪽 도크 점유 상태로 시작** → 부팅 직후 복귀 로봇이 점유 도크로 도킹하는 충돌 차단.
- **맵 변경 대응:** 좌측 복도(Traffic_L) 제거에 맞춰 그래프 토폴로지 재설계.

### 3. 오프라인 로봇 배정 차단 — "실패가 시스템을 멈추지 않게" *(RobotStateMonitor)*
- **문제:** bringup이 안 켜진 로봇이 **오래된 배터리값(시드 100%)** 때문에 주문을 배정받고 이동 단계에서 예외로 빠짐.
- **해결:** RobotStateMonitor가 로봇별 마지막 배터리 수신 시각을 추적, **20초 미수신 시 0%로 강제** → 기존(이명제 담당 TaskManager) "배터리≤30% 배정 제외" 게이팅이 자동 차단하도록 **내 모듈 쪽에서 신호를 만들어 연동**.

### ★ 대표 설계 — Reverse Docking 정밀 후진 도킹 *(PICKY 주행, 박서우 단독 설계)*
좁은 충전 도크에 후진 삽입. **"코드보다 정밀도 한계를 먼저 못 박은"** 설계 추론이 핵심.

- **정밀도 제약을 기하로 역산:** 후진 0.33 m에서 횡오차 1 cm 이내 → `0.01 = 0.33·sin θ` ⇒ **`yaw ≲ 1.7°`** 가 모든 설계 선택을 지배.
- **병목 진단:** 단일 마커 solvePnP는 **lateral은 쉽지만(1 px ≈ 0.5 mm)** **yaw가 취약(1~3° + pose-flip ambiguity)** — 하필 그 yaw가 제약.
- **끝까지 추적한 두 근본원인:**
  - ① **카메라 fx 가 1.48배 작았다** — 정지 진단툴로 로봇을 아는 위치에 두고 깊이만 바꿔 측정하니, `tx`(횡)는 fx 무관하게 정확(마커간격 0.174 vs 실제 0.170)한데 `tz`(깊이)는 fx 비례라 짧게(실측 D/tz=1.48) 나왔다. `depth_scale 1.36`·`lateral_scale 1.48` 은 진짜 스케일이 아니라 **틀린 fx 를 거리마다 덮던 fudge**였다(끝없는 부호 튜닝이 수렴 못 한 이유). → `lateral_scale` 1.48→1.0 으로 횡 1.8cm 과보정 제거.
  - ② **단일 평면 마커론 헤딩이 구조적 병목**(pose-flip ±5~8°, 같은 벽인데 두 마커가 4° 불일치). 도크의 **마커 2개**를 쓰면 두 기지점의 **translation 만**(노이즈 큰 rvec 미사용) 강체정합해 로봇 yaw 를 **ambiguity 없이**(σ±0.05°) 얻는다.
- **최종 설계(두 마커 localization + odom 정밀 실행):**

  | 축 | 담당 (최종) | 방식 |
  |---|---|---|
  | **헤딩(yaw)** | **두 마커 translation** | `psi_two=atan2(Δtz·fx, Δtx)`, pose-flip 없음 |
  | **횡·위치** | 두 마커 강체정합 → **odom 곡선 이동** | robot(x,y) 1회 측정 → (dock_x, approach_y, 법선)으로 정밀 주차 |
  | depth · 정지 | ArUco 마커 | 월드 깊이 `robot_y ≤ dock_y` |
  | 위치보정 | ArUco 마커 | 도킹 후 `/initialpose`(AMCL 재초기화) |

- **6단계 시퀀스:** ①주마커 정렬 ②법선 yaw 정렬 ③10cm 후진(두-마커 영역 진입) ④쌍마커 검출+월드 pose 측정(가까워 partner 화각밖이면 2cm씩 추가후진 재시도) ⑤odom 정밀 곡선 이동(x=dock_x, y=0.18, yaw=법선) ⑥dock_y 까지 직진 후진. `reverse_dock()` 시그니처 불변(state_manager 무수정).

- **단일 vs 두 마커 (실측):**

  | 항목 | 단일 마커 | 두 마커 |
  |---|---|---|
  | 헤딩 측정 노이즈 | ±3~8° (pose-flip) | **±0.05°** |
  | 깊이 의존 x 편향 | ±1.4cm/깊이 | **없음** |
  | 최종 도킹 x | ~1cm(0.12) | **~3~5mm(0.11)** |
  | 최종 도킹 yaw | 비뚤(2~3°) | **곧음(<1°)** |
  | 좁은 도크 도킹 | 마진 부족 | **안정 통과** |

> **개발자 코멘트** — *기하로 못 박은 제약(yaw≲1.7°)은 끝까지 옳았다. 핵심은 "추측으로 부호·스케일을 튜닝"하는 대신 **정지 진단툴로 fx 오류와 pose-flip 을 숫자로 격리**한 것. 단일 마커는 횡은 쉬워도 yaw 가 병목이었고, **두 기지점의 translation 으로 헤딩을 ambiguity 없이 잡고 odom 으로 정밀 실행** — 문제를 게인이 아니라 구조로 풀었다.*

## 슬라이드 3 — 검증·트러블슈팅과 부가 기여

### 정량 검증 (내 담당 범위)
| 항목 | 내용 |
|---|---|
| 단위 테스트 | **TrafficManager 70개 통과** (경로 예약·도크 점유·콜드스타트) |
| Nav2 튜닝 | inflation 0.08m · 정밀접근 정지 2cm · RPP lookahead 0.15~0.45 · 코스트맵 해상도 3cm |
| 좌표 정합 | 맵 origin을 SLAM 시작 오프셋 반영해 `[-0.08,-0.12]`로 정합(map=arena 절대좌표), amcl 초기 위치를 충전 도크에 정합 |
| 통신 | `ROS_DOMAIN_ID=25` 팀 통일, 텔레메트리 ROS2 토픽 단일화(HTTP 보고 폐기) |

### 대표 디버깅 (근본 원인 추적)
- **CPU 폭주:** `/tf` 구독 노드를 한 executor에 묶어 wait-set 재구성 busy-wait → **노드별 executor 분리**로 해소.
- **재시작 복구(R1):** 노드 재기동 시 로봇 현재 위치로 점유만 복원 + 텔레메트리로 완료 재동기.
- **amcl 위치추정 실패:** initial_pose를 리스트로 주면 무시됨 → `x/y/yaw` 서브키 형식으로 수정.
- **RViz "queue full":** map→odom 변환 미수신이 원인 → namespace TF remap·bringup 완료 후 RViz 기동으로 정리.

### 실로봇 주행 정밀도 튜닝 — 근본원인 추적 (2026-06)
picky1 단독 실주행을 반복하며 "도착 오판 · 코너 충돌 · 매대 오정렬 · navigation abort"를 하나씩 **근본 원인까지 추적**해 해결. 증상이 아니라 원인을 고친 사례.

| 증상 | 근본 원인 | 해결 | 성과 |
|---|---|---|---|
| 도착 전 "도착" 오판 + 벽 충돌 | `map.yaml origin=[0,0,0]`로 맵 프레임이 arena 절대좌표와 (0.06,0.12) 어긋남 | origin `[-0.08,-0.12]` 복원 (odom 1m=1m 검증해 스케일 가설 배제) | 좌표 정합·조기 도착 해소 |
| 코너에서 경유지 질러 충돌 | 소형 아레나 대비 RPP lookahead 과대(실효 0.3m) | lookahead 0.6→0.25 / 0.3→0.15 / 0.9→0.45 | 코너 cut 제거 |
| 진입 heading 기울어 매대 오정렬 | TRAFFIC 노드 x가 PRODUCT_ZONE과 미정렬 | 같은 열 x 정렬(0.64/1.05/1.48) | 순수 수직 진입 |
| 매대 5cm 앞 정지 | move_to_goal 자체 tolerance 0.05(Nav2와 별개) | 0.02 + 근거리 조향 가드 | 2cm 접근·떨림 없음 |
| navigation abort | 보드 CPU 과부하(load 20+)로 planner ack 20ms 초과 | server_timeout 20→100ms + update_min_d 0.05→0.10 | abort 완화·부하 경감 |
| 도착 후 매대 바라보고 주차 | nearest-90 스냅이 수평 진입 때 오정렬 | task_type별 정지 자세 정책(법선/±x/+y) | 진입 무관 사이드 주차 |

> **핵심:** odom · AMCL · 맵 프레임 · costmap · 컨트롤러를 분리 진단해, 같은 증상(벽 충돌)의 서로 다른 원인 5가지를 각각 격리·수정했다.

### Reverse Docking 실차 디버깅 — 근본원인 추적 (2026-06)
설계를 실차에 올리며 "마커 미검출 / 후진 중 엉뚱한 회전"을 단계적으로 근본원인까지 추적.

| 증상 | 근본 원인 | 해결 |
|---|---|---|
| 30s acquire timeout(마커 0회 검출) | 보드에 `camera/image_raw` 발행 노드 없음(ROS 토픽 의존 설계의 빈 발행자) | reverse_docking 이 **Picamera2 직접 캡처**(도킹 중에만) |
| 카메라 켜도 마커 미검출 | 도크 마커는 **AprilTag 36h11** 인데 코드는 ArUco 4x4_50 으로 탐색 | dict 정정 + 캘리브 yaml 직접 로드 |
| 깊이(정지시점) 어긋남 | marker_size 0.10 인데 실측 0.05 | size 0.05 |
| 후진 중 엉뚱한 라인 인식 | 주차선 3줄(중앙 공유)인데 좌/우 절반 평균 피팅 | 컬럼 히스토그램 분리 → dock별 채널 2줄 선택 |
| 정렬이 반대로 → 벽 충돌 | **카메라 거꾸로 장착 + flip 캘리브레이션**인데 flip 미적용 → 마커 횡/yaw 부호 반전 | 검출 전 `cv2.flip(-1)` (팀 perception 과 일치) |

**2차(제어 아키텍처) — 검출은 됐는데 제어가 발산/드리프트:**

| 증상 | 근본 원인 | 해결 |
|---|---|---|
| 횡 제어 발산(법선 지나쳐 아크로 멀어짐) | 실시간 `robot_x` 가 **노이즈 psi 에 종속** — 회전만 해도 robot_x 가 수 cm 헛돔(부호로 해결 불가) | 실시간 추적 폐기 → **마커 1회 측정 + odom arc** |
| 측정 robot_x 부정확(조준 ≠ 법선) | 마커 "조준"(tvec_x→0)은 헤딩 틀어진 채로도 성립 → robot_x 가 헤딩과 결합 | psi 디커플링 + **측정 전 똑바로 세움**(psi≈0 → Δx 깨끗) |
| 횡오차 1.48배 과소 | Δx 에 solvePnP 스케일 누락(깊이엔 적용, 횡엔 raw) | 깊이/횡 스케일 분리(depth 1.36 / lateral 1.48) |
| 도크 4cm 앞 조기 정지 | depth_scale 가 가까운 거리(standby)서 튜닝 → 원거리(dock) 과대 | 원거리 스케일 1.48→1.36 |
| 후진 중 yaw 누적 드리프트 | 작은 yaw × 후진거리 = x 드리프트, psi 는 노이즈 | **odom yaw 앵커 + 후진 내내 유지**(`yaw_drift` ±0.2°) |

**3차(정밀도 완성) — 정지 진단툴로 숨은 원인 격리:**

| 증상 | 근본 원인 | 해결 |
|---|---|---|
| 같은 x 인데 깊이마다 x 가 다름 | **카메라 fx 가 1.48배 작음**(영상모드 crop≠캘리브) → tz·회전 오염. `tx`는 fx 무관(정확)인데 `depth/lateral_scale`이 그걸 덮던 fudge | 진단툴 fx sweep 로 D/tz=1.48 확정. `lateral_scale` 1.48→1.0(횡 1.8cm 과보정 제거) |
| 정렬돼도 후진이 비뚤음 | 단일 평면 마커 yaw 의 구조적 한계(pose-flip, 두 마커가 4° 불일치) | **두 마커 translation 강체정합**으로 헤딩 ambiguity 없이(σ±0.05°) 확보 → odom 곡선 이동 |
| 가까우면 두-마커 검출 실패 | 가까울수록 두 마커 각이 커 partner 가 화각밖 | 검출 실패 시 **2cm씩 추가후진 재시도**(멀어지며 각 좁아짐) |
| 두-마커 x 살짝 서쪽 과진입 | odom 곡선 횡이동 미세 과보정 | 카메라 offset 대신 **곡선 횡델타 0.9배**(다른 경로 무영향) |

> **핵심:** 1차는 카메라 파이프라인(발행자→dict→크기→flip), 2차는 rvec yaw 노이즈로 "마커 1회+odom" 전환, 3차는 **정지 진단툴로 fx 오류·pose-flip 을 숫자로 격리**해 두-마커로 완성. *추측 튜닝 대신 "아는 위치에서 측정"으로 숨은 계통오차를 잡고, 문제를 게인이 아니라 구조(두 기지점 translation)로 풀었다.*
